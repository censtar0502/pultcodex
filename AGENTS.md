# AGENTS.md

## Purpose

This repository contains firmware for an operator panel controlling two independent fuel dispensers.

The current development platform is:

- `STM32F407G-DISC1`

The target custom board is currently planned around:

- `STM32F407VET6`

Primary protocol direction:

- `GasKitLink` first
- later: `AZT 2.0`, `Tatsuno`, optionally `SANKI`

## Build and safety rules

- The project must remain `STM32CubeIDE` / `CubeMX` safe.
- Do not casually modify autogenerated code outside `USER CODE BEGIN/END` regions.
- If Cube-generated files must be adjusted, keep the change minimal and verify it survives regeneration.
- Prefer minimal, localized patches.
- Keep the project buildable after each meaningful change.

## Firmware rules

- Avoid new blocking behavior.
- Do not add `HAL_Delay`, `HAL_MAX_DELAY`, busy-wait loops, or equivalent blocking calls without explicit user permission and a concrete justification.
- Prefer static memory over dynamic allocation.
- Do not add new dependencies without a clear reason.
- Do not log to USB directly from ISR context.
- ISR should do the minimum possible: capture data, set flags, push to buffers.
- Main loop / task layer should handle parsing, UI updates, and logging.

## UART / dispenser rules

- Treat `TRK1` and `TRK2` as fully independent logical channels.
- Never duplicate source trees for each TRK.
- Use separate runtime contexts instead.
- Protocol-specific frame formats, CRC/XOR, status codes, and command bytes must stay inside protocol adapters.
- FSM must not depend on raw protocol frames.
- Current physical exchange direction is based on `9600 8N1` unless documented otherwise for a specific protocol.

## Display and UI rules

- Current OLED is `SSD1322 SPI`.
- For SSD1322:
  - opcode byte is sent with `DC=LOW`
  - parameter/data bytes are sent with `DC=HIGH`
- On the current dev stand, `SPI_DIRECTION_2LINES` is required for correct OLED operation.
- UI should react to keyboard `DOWN` events, while raw logs may still retain both `DOWN` and `UP`.

## Logging rules

- USB CDC is the current service log channel.
- Future dispenser traffic logging must support:
  - `TRK1`
  - `TRK2`
  - `BOTH`
  - `OFF`
- These logging controls belong in code/service logic, not in operator UI.

## Hardware awareness

- `DISC1` pinout is temporary and may differ from the future custom PCB.
- Do not assume dev-board UART or GPIO placement is final hardware architecture.
- Re-check any pin-sensitive change against:
  - `.docs/DISC1_DEV_PINOUT.md`
  - `.docs/HARDWARE_AND_PINOUT_F407.md`

## Expected output style for changes

- Report only the important changed files and behavior.
- Mention residual risks clearly.
- If build/test could not be run, say so explicitly.
